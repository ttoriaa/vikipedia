# 数据产品 PRD 标准模板

## 0. 使用说明

这份模板面向数据产品经理，用于标准化从业务问题到可落地产品方案的完整 PRD。适用对象包括数据看板、指标平台、经营分析平台、智能分析助手、数据中台模块、数据质量平台、数据工作流工具、AI + Data 应用等。

写 PRD 时不需要机械填满所有章节，但必须回答四个核心问题：为什么做、给谁用、怎么用、如何验证成功。对于不确定内容，用 `Assumption` 标注，并进入 Open Questions。

## 2. 需求摘要

需求摘要是 PRD 的第一页结论区，目标是让业务方、产品、设计、研发、数据和测试在 3-5 分钟内理解：这个产品为什么值得做、服务谁、解决什么问题、成功如何衡量、当前还有哪些关键不确定性。

写这一章时遵循三个原则：

- 先讲问题和价值，再讲功能；PRD 不是功能堆叠清单，而是围绕用户任务和业务目标形成的对齐文档。
- 保持足够具体；不要写“提升效率”这类无法验证的表达，要写清楚提升谁的什么效率、从多少提升到多少、在哪个时间窗口验证。
- 显式标注假设、约束和依赖；任何尚未被数据、访谈或业务 owner 确认的内容，都要进入 Open Questions 或 Assumption。

### 2.1 一句话定义

用一句话说明这个产品解决什么问题、服务谁、产生什么价值。建议使用以下句式：

```text
为 [目标用户 / 业务角色] 提供 [产品形态 / 核心能力]，帮助其在 [关键场景] 中解决 [核心问题]，从而实现 [可衡量业务结果]。
```

一句话定义必须包含四个要素：

| 要素 | 要回答的问题 | 数据产品写法提示 | 示例 |
|---|---|---|---|
| 目标用户 | 谁最需要这个产品？ | 写具体角色，不写泛泛的“业务用户” | 销售管理层、区域运营、数据分析师 |
| 关键场景 | 用户在哪个工作流里使用？ | 写会议、复盘、监控、归因、决策等场景 | 每日经营例会、月度复盘、异常处理 |
| 核心问题 | 当前最痛的阻塞是什么？ | 指向数据、指标、流程或决策痛点 | 指标口径不一致、异常定位慢、行动不可追踪 |
| 业务结果 | 做完以后什么会变好？ | 尽量量化，并能被验收 | 分析耗时下降 50%，异常闭环率提升到 80% |

推荐示例：

- 为销售管理层提供一个跨区域、跨渠道、跨产品线的经营分析平台，帮助管理者在每日例会上快速识别异常、定位原因并形成行动闭环。
- 为区域运营团队提供一个指标监控与异常归因工具，帮助其在销量、库存、转化率波动时自动定位影响维度，并把处理动作沉淀为可追踪任务。
- 为数据分析师和业务 owner 提供一个统一指标管理平台，帮助团队标准化指标口径、追踪指标血缘和变更记录，降低重复取数和口径争议。
- 为管理层和一线执行团队提供一个 AI 辅助经营分析助手，帮助用户用自然语言查询核心指标、理解变化原因，并生成可验证的数据引用和行动建议。

不推荐写法：

- 做一个数据平台，提高业务效率。
- 做一个智能看板，让大家看到数据。
- 做一个 AI 分析系统，帮助公司智能决策。

原因：这些表达缺少目标用户、使用场景、核心问题和衡量标准，研发、设计和测试无法据此判断范围和验收方式。

### 2.2 问题定义

问题定义用于把“想做一个产品”转化为“有证据的问题陈述”。推荐用 `现状 -> 影响 -> 根因 -> 机会 -> 不做的后果` 的方式组织。

| 维度 | 内容 |
|---|---|
| 当前问题 | 描述用户今天遇到的具体阻塞，不直接写解决方案。例如：核心经营指标分散在多个系统和 Excel 中，口径不一致，管理层难以在例会前获得可信结论。 |
| 影响对象 | 明确受影响的角色、团队和决策链路。例如：销售管理层、区域运营、数据分析师、财务 BP、一线负责人。 |
| 影响范围 | 描述涉及的业务范围、组织范围、数据范围和频率。例如：覆盖 8 个区域、3 条产品线、每日经营例会和月度复盘。 |
| 现有做法 | 描述当前流程和工具。例如：分析师从 CRM / ERP / 埋点系统导出数据，手工合并 Excel，再制作 PPT 给管理层。 |
| 痛点表现 | 慢、错、散、不可追溯、重复劳动、指标口径不一致、异常无人认领、结论难以复用。 |
| 业务损失 | 时间、人力、成本、风险、机会损失。例如：每周重复制作报表 20 小时，异常从发现到定位平均超过 2 天。 |
| 根因假设 | 初步判断问题产生的底层原因。例如：缺少统一指标字典、缺少数据质量门禁、缺少权限隔离、缺少异常闭环机制。 |
| 为什么现在做 | 业务窗口、组织变化、技术成熟、合规压力、竞争压力。例如：业务规模扩大后手工流程不可持续，管理层要求统一经营口径。 |
| 不做的后果 | 如果不做会发生什么。例如：决策继续依赖离线表格，跨团队争议增加，异常响应滞后，数据资产无法沉淀。 |

问题定义建议补充三类证据：

| 证据类型 | 可收集材料 | 用途 |
|---|---|---|
| 定量证据 | 报表制作耗时、数据延迟、错误率、异常处理时长、重复取数次数 | 证明问题真实存在并可衡量 |
| 定性证据 | 用户访谈、会议纪要、业务投诉、工单记录、复盘材料 | 解释用户为什么痛、痛在哪里 |
| 系统证据 | 数据源清单、接口现状、权限规则、指标口径文档、历史 SQL / Excel | 判断技术可行性和实施复杂度 |

问题陈述模板：

```text
当前 [目标用户] 在 [业务场景] 中，需要通过 [现有做法] 完成 [关键任务]。
但由于 [根因假设]，导致 [痛点表现]，进而造成 [业务损失]。
如果不解决，预计会影响 [影响范围 / 决策链路 / 风险后果]。
因此，本产品需要优先解决 [最核心问题]，并通过 [成功指标] 验证价值。
```

数据产品示例：

```text
当前销售管理层在每日经营例会前，需要通过分析师手工整理 CRM、ERP 和渠道报表来判断销量、库存和转化率变化。
但由于核心指标口径分散、数据刷新延迟不可见、异常缺少统一归因入口，导致管理者经常在会议中讨论“数据是否可信”，而不是直接讨论业务动作。
这会造成报表制作重复投入、异常响应滞后、跨区域复盘不可追溯。
因此，本产品需要优先解决核心指标统一展示、异常自动识别和行动闭环追踪，并通过报表制作耗时下降、异常确认率和行动闭环率验证价值。
```

问题定义完成后，应能回答以下判断题：

- 这个问题是否足够高频，值得产品化，而不是一次性分析项目？
- 是否有明确 owner 对结果负责？
- 问题是否能通过数据或流程变化被验证？
- 是否区分了表层现象和底层根因？
- 是否说明了不做的业务代价？
- 是否已经避免把“我要一个看板”误写成问题本身？

### 2.3 需求完整度评估

需求完整度评估用于判断 PRD 是否可以进入方案设计和开发排期。低分不是问题，真正的风险是低分却没有显式标注假设和补充调研计划。

| 维度 | 分数 | 评分标准 | 需要补充的材料 |
|---|---:|---|
| 业务目标清晰度 |  / 20 | 0-5：只有方向；6-10：有目标但无指标；11-15：有 KPI 但无基线；16-20：有 KPI、基线、目标值和时间窗口 | OKR、KPI、业务复盘、管理层目标 |
| 用户与场景清晰度 |  / 20 | 0-5：用户泛化；6-10：有角色但无任务；11-15：有核心场景；16-20：有用户旅程、频率、优先级和验收场景 | 用户访谈、角色清单、流程图、UAT 场景 |
| 数据源清晰度 |  / 20 | 0-5：不知道数据源；6-10：知道系统但不了解字段；11-15：有字段和刷新频率；16-20：有质量规则、owner、权限和历史问题 | 数据源清单、数据字典、样例数据、质量报告 |
| 功能边界清晰度 |  / 20 | 0-5：功能愿望清单；6-10：有模块但无优先级；11-15：区分 MVP / V1 / V2；16-20：有 In Scope / Out of Scope 和验收标准 | 功能清单、优先级、原型、版本计划 |
| 技术约束清晰度 |  / 20 | 0-5：无约束；6-10：有技术偏好；11-15：有性能、安全、集成要求；16-20：有 SLA、权限、审计、部署、回滚和依赖系统说明 | 架构约束、SSO / API 文档、安全合规要求、运维规范 |
| 总分 |  / 100 | 低于 70 分建议先补充调研；70-84 可进入 MVP 方案设计；85 分以上可进入开发排期评审 | 完整 PRD、原型、技术方案、测试计划 |

评分解释：

| 总分区间 | 判断 | 建议动作 |
|---:|---|---|
| 0-49 | 需求仍处于想法阶段 | 先做问题澄清、用户访谈和数据源盘点，不建议进入开发 |
| 50-69 | 可以做方案探索，但风险较高 | 产出 PRD V0.1、低保真原型和关键假设验证计划 |
| 70-84 | 可以进入 MVP 设计 | 明确 P0 功能、接口草案、数据模型和 UAT 场景 |
| 85-100 | 可以进入排期评审 | 准备研发排期、测试计划、上线验收和回滚方案 |

需求摘要输出物建议：

| 输出物 | 目的 | 最低完成标准 |
|---|---|---|
| 一句话定义 | 对齐产品方向 | 包含目标用户、场景、问题、价值 |
| 问题定义表 | 证明问题值得解决 | 至少有现状、影响对象、业务损失、为什么现在做 |
| 证据清单 | 支撑 PRD 可信度 | 至少有 1 类定量证据和 1 类定性证据 |
| 需求完整度评分 | 判断能否进入下一阶段 | 每项给分并说明扣分原因 |
| Open Questions | 管理未知项 | 每个问题有 owner、截止时间和影响范围 |

需求摘要评审 Checklist：

- 是否能让非产品同事在 3 分钟内理解为什么要做？
- 是否避免把解决方案当成问题，例如“缺一个看板”？
- 是否明确首批用户，而不是覆盖所有人？
- 是否有可量化的成功指标或至少有待补充的基线计划？
- 是否说明数据产品特有问题：数据源、口径、刷新、质量、权限、审计？
- 是否写清楚当前阶段不做什么，避免范围膨胀？
- 是否列出关键假设、约束和依赖，方便后续评审和排期？

## 3. 背景与机会

背景与机会用于回答“为什么现在值得做”。这一章不是简单介绍公司或行业，而是把业务环境、组织目标、用户需求、现有流程、市场/竞品信息和机会判断放在同一个上下文里，帮助团队决定是否进入方案设计与投入评审。

写这一章时建议遵循 `业务背景 -> 市场/竞品背景 -> 机会判断 -> 关键假设` 的顺序。对于内部数据产品，“市场”不一定是外部商业市场，也可以是内部服务对象、可覆盖流程、可节省人力、可降低风险和可沉淀的数据资产。

### 3.1 业务背景

业务背景需要说明产品所在的业务上下文，以及它如何支撑组织目标。不要只写“公司需要数字化”这类抽象表述，而要写清楚：哪个业务域、哪条流程、谁负责、当前用什么系统、哪里出现了规模化瓶颈。

| 字段 | 写法说明 | 示例 |
|---|---|---|
| 所属业务域 | 明确产品服务的业务范围，例如销售、售后、供应链、财务、内容运营、风控、制造质量等 | 销售经营分析 / 区域运营管理 / 数据质量治理 |
| 业务现状 | 描述当前业务规模、流程复杂度、数据体量、组织协作方式和主要矛盾 | 覆盖 8 个区域、3 条产品线，管理层每周需要查看销售、库存、转化、回款等核心指标 |
| 组织目标 | 对齐公司、部门或项目 OKR，说明产品如何服务增长、降本、提效、合规、风险控制或体验提升 | 将经营例会从“解释数据”转向“决策行动”，提升异常响应速度 |
| 关键利益相关方 | 区分 sponsor、业务 owner、核心用户、数据 owner、研发 owner、安全/合规 owner | Sales VP、区域运营负责人、BI 团队、数据平台团队、信息安全团队 |
| 现有系统和流程 | 描述当前工具链和用户路径，标注系统边界、人工节点和重复劳动 | CRM 导出客户数据，ERP 导出订单数据，Excel 合并后由分析师制作 PPT |
| 当前基线 | 给出可对比的现状指标，便于后续衡量产品效果 | 报表制作每周 20 小时；异常定位平均 2 天；核心指标口径争议每月 5 次 |
| 约束条件 | 描述时间、人力、预算、数据、系统、安全、上线窗口等限制 | MVP 需要 8 周内上线；必须接入企业 SSO；财务数据需要脱敏 |

业务背景建议补充一张“业务上下文画布”：

| 画布项 | 需要回答的问题 | 产出物 |
|---|---|---|
| 业务目标 | 这项工作支持哪个战略目标或 OKR？ | 目标-指标映射表 |
| 核心流程 | 产品嵌入哪条业务流程？流程起点和终点是什么？ | 当前流程图 / 用户旅程图 |
| 关键决策 | 用户希望用产品做什么判断或行动？ | 决策清单 / 例会场景 |
| 数据资产 | 决策依赖哪些数据源、指标和维度？ | 数据源清单 / 指标字典 |
| 协作机制 | 哪些团队会共同使用或维护？责任如何分配？ | RACI / owner 清单 |
| 成功变化 | 产品上线后，哪些行为或指标应该变化？ | 基线与目标值 |

关键利益相关方可以用 RACI 方式管理：

| 角色 | 代表人 / 团队 | 责任类型 | 关注点 | 需要参与的评审 |
|---|---|---|---|---|
| Sponsor |  | Accountable | 投入产出、业务价值、优先级 | 立项、范围冻结、上线验收 |
| 业务 Owner |  | Responsible | 业务流程、指标口径、行动闭环 | PRD、UAT、复盘 |
| 核心用户 |  | Consulted | 使用体验、任务完成效率 | 用户访谈、原型评审、UAT |
| 数据 Owner |  | Responsible | 数据源、质量、口径、血缘 | 数据模型、指标字典、质量规则 |
| 技术 Owner |  | Responsible | 架构、接口、性能、稳定性 | 技术方案、接口评审、上线准备 |
| 安全 / 合规 |  | Consulted / Approver | 权限、审计、脱敏、合规边界 | 安全评审、上线审批 |

现有系统和流程建议不要只列系统名，而要画出“数据如何流动、用户如何工作”：

| 流程节点 | 当前做法 | 使用系统 / 文件 | 责任人 | 耗时 | 问题 | 可产品化机会 |
|---|---|---|---|---:|---|---|
| 数据获取 | 从多个系统导出数据 | CRM / ERP / 埋点 / Excel | 分析师 |  | 手工导出、字段不一致 | 自动接入、标准化字段 |
| 数据处理 | Excel / SQL 清洗合并 | Excel / SQL / BI | 分析师 |  | 容易出错、不可追溯 | 数据模型、质量校验 |
| 指标计算 | 手工计算或复制历史 SQL | SQL / Excel | BI |  | 口径版本混乱 | 指标字典、指标服务 |
| 结果呈现 | 制作 PPT 或静态报表 | PPT / BI | 分析师 |  | 更新慢、难下钻 | Dashboard、交互分析 |
| 业务行动 | 群消息分派任务 | IM / 邮件 | 运营负责人 |  | 无状态跟踪 | 行动项、闭环工作流 |

业务背景完成后，应能回答：

- 这个产品和组织目标之间的关系是什么？
- 当前流程是否真的存在规模化瓶颈？
- 首批业务 owner 是否愿意投入时间参与评审和 UAT？
- 是否有清晰的现状基线，方便上线后验证效果？
- 是否有数据、系统、安全或组织约束会影响 MVP 范围？

### 3.2 市场 / 行业 / 竞品背景

市场 / 行业 / 竞品背景用于说明：外部趋势、内部旧系统、用户替代方案和竞品能力，如何影响本产品的定位和边界。对于数据产品，竞品不只包括商业 SaaS，也包括内部 BI、Excel、SQL 脚本、自动化报表、数据中台模块和人工流程。

调研建议围绕五个问题展开：

1. 目标市场或内部服务对象是谁，为什么值得投入？
2. 用户和买单方是否一致？如果不一致，分别关心什么？
3. 用户当前用什么替代方案完成任务？替代方案哪里不足？
4. 行业内成熟工具提供了哪些能力，哪些不适合当前组织？
5. 我们要差异化解决什么问题，而不是重复造一个普通看板？

| 类型 | 调研对象 | 重点问题 | 发现 | 对本产品的启发 |
|---|---|---|---|
| 行业趋势 | 行业报告、咨询报告、技术趋势、政策法规、头部企业实践 | 数据产品正在从报表展示走向指标治理、自动洞察、AI 问数和行动闭环吗？ |  | 判断产品是否顺应组织数字化、智能化、治理化方向 |
| 内部竞品 / 旧系统 | 现有 BI、Excel 报表、数据门户、数据中台、历史脚本、运营台账 | 用户为什么继续使用旧工具？旧工具的边界和沉没成本是什么？ |  | 避免重复建设，明确迁移、集成或替代策略 |
| 外部竞品 | Power BI、Tableau、Looker、Amplitude、Mixpanel、dbt Semantic Layer、DataHub、Monte Carlo 等同类能力 | 它们解决了什么问题？强项、弱项、使用门槛和成本是什么？ |  | 借鉴成熟能力，识别不适合照搬的地方 |
| 用户访谈 | 管理层、运营、分析师、一线执行、数据 owner | 用户真实任务是什么？现在最常用的替代方案是什么？为什么痛？ |  | 校准核心场景和优先级，避免只从技术视角定义产品 |
| 数据分析 | 使用日志、工单、报表下载、SQL 查询、会议材料、异常处理记录 | 哪些指标最高频？哪些流程最耗时？哪些错误最常见？ |  | 用证据支撑机会判断和 MVP 范围 |

外部竞品可以按能力层而不是品牌名称拆解：

| 竞品 / 替代方案类型 | 代表能力 | 优势 | 局限 | 对本产品的启发 |
|---|---|---|---|---|
| 传统 BI 工具 | 报表、Dashboard、可视化、权限 | 成熟稳定，适合标准报表 | 对指标治理、异常闭环和业务动作支持有限 | 不只做展示层，要补足指标口径、归因和闭环 |
| 产品分析工具 | 用户行为分析、漏斗、留存、分群 | 适合数字产品增长分析 | 对企业经营、财务、供应链等复杂数据源支持有限 | 借鉴自助分析和事件模型，但需适配企业指标体系 |
| 指标 / 语义层工具 | 指标定义、口径复用、语义查询 | 降低口径冲突 | 往往需要较强数据工程基础 | 指标字典和版本管理应作为核心能力 |
| 数据目录 / 治理工具 | 数据资产、血缘、owner、质量 | 有利于治理和合规 | 离业务决策链路较远 | 元数据要和实际业务页面连接，而不是孤立目录 |
| 数据质量 / 可观测性工具 | 质量规则、告警、血缘影响分析 | 提升可信度 | 不能直接解决业务行动闭环 | 质量状态要在用户使用场景中可见 |
| AI 分析助手 | 自然语言问数、总结、解释 | 降低使用门槛 | 容易出现幻觉、越权和不可追溯 | 必须强制引用数据来源、权限校验和人工确认 |
| 人工流程 / Excel | 灵活、低成本、上手快 | 适合早期探索 | 难规模化、难审计、难复用 | MVP 要保留灵活性，同时把高频动作产品化 |

### 3.3 机会判断

机会判断用于决定是否值得进入 MVP。建议把机会拆成“用户价值、业务价值、可行性、时机、风险”五类，而不是只凭直觉判断。

| 判断项 | 关键问题 | 评分 1-5 | 结论 / 证据 |
|---|---|---:|---|
| 是否高频 | 用户是否每天 / 每周都会遇到？是否嵌入固定业务节奏？ |  |  |
| 是否高价值 | 是否影响收入、成本、效率、风险、合规或关键决策？ |  |  |
| 是否可规模化 | 是否可以从一个团队扩展到多个区域、业务线或角色？ |  |  |
| 是否能用数据闭环验证 | 是否有基线、目标值和上线后可追踪指标？ |  |  |
| 是否有明确 owner | 是否有业务 owner、数据 owner 和技术 owner？ |  |  |
| 是否适合当前阶段投入 | MVP 是否能在当前时间、人力、数据和系统约束下完成？ |  |  |
| 用户迁移成本是否可控 | 用户从旧流程迁移是否需要大量培训、流程改造或组织推动？ |  |  |
| 差异化是否清晰 | 是否明确不同于旧 BI、Excel 或外部工具的核心价值？ |  |  |

评分建议：

| 分数 | 含义 |
|---:|---|
| 1 | 证据不足或价值很弱，不建议投入 |
| 2 | 有局部价值，但用户、数据或 owner 不清晰 |
| 3 | 有明确场景，可以继续调研或做原型验证 |
| 4 | 价值明确，可进入 MVP 规划 |
| 5 | 高频高价值且 owner 明确，应优先投入 |

可以使用轻量加权模型做优先级判断：

| 维度 | 权重 | 评分 | 加权分 | 说明 |
|---|---:|---:|---:|---|
| 用户频率 | 15 |  |  | 使用越高频，越适合产品化 |
| 业务影响 | 25 |  |  | 对收入、成本、风险或决策影响越大，权重越高 |
| 数据可得性 | 15 |  |  | 数据源越稳定，MVP 风险越低 |
| 解决方案可行性 | 15 |  |  | 技术、组织、流程越可控，越适合推进 |
| 可规模化 | 10 |  |  | 能否复用到更多业务单元 |
| 可验证性 | 10 |  |  | 是否能设定基线和成功指标 |
| owner 明确度 | 10 |  |  | 是否有人负责推动、验收和运营 |
| 总分 | 100 |  |  | >= 75 建议进入 MVP；60-74 先做验证；< 60 暂缓 |

机会规模可以用内部数据产品的方式估算：

| 估算项 | 公式 / 方法 | 示例 |
|---|---|---|
| 覆盖用户数 | 目标角色人数 x 使用频率 | 50 名区域运营，每周使用 3 次 |
| 节省时间 | 当前单次耗时 - 目标单次耗时 | 单次分析从 2 小时降到 30 分钟 |
| 年化效率收益 | 节省时间 x 使用频率 x 用户数 x 人力成本 | 用于粗略 ROI 估算 |
| 风险降低 | 错误率 / 延迟 / 越权 / 合规问题下降 | 指标口径争议减少 50% |
| 决策改善 | 异常发现提前量、行动闭环率、复盘完成率 | 异常从 T+2 定位提前到 T+0 |

机会判断输出结论建议使用以下格式：

```text
基于当前证据，该机会属于 [高 / 中 / 低] 优先级。
优先级判断依据是：[高频场景]、[业务影响]、[数据可得性]、[owner 明确度]。
建议下一步：[进入 MVP 方案设计 / 先做原型验证 / 先补充调研 / 暂缓]。
进入下一阶段前必须确认：[关键假设 1]、[关键假设 2]、[关键依赖]。
```

机会判断评审 Checklist：

- 是否说明这是内部效率工具、经营决策平台、数据治理产品还是商业化产品？
- 是否区分了用户、买单方、维护方和审批方？
- 是否比较了内部旧系统、人工替代流程和外部竞品？
- 是否能解释为什么不用现成 BI / Excel / 外部 SaaS 就能解决？
- 是否有清晰的 MVP 切入点，而不是一次性做完整平台？
- 是否有上线后验证路径，包括使用、效率、质量和业务结果？
- 是否明确哪些假设一旦不成立，项目应该降级或暂停？

## 4. 目标与成功标准

目标与成功标准用于回答“上线以后如何判断产品真的成功”。这一章要把业务目标、用户价值、产品使用、数据质量、技术稳定性和风险控制连接起来，形成可衡量、可追踪、可复盘的 KPI 体系。

好的目标不是口号，而是能指导取舍：当团队面对范围、体验、技术方案和上线节奏冲突时，可以用这些目标判断什么更重要。建议遵循 SMART 原则：具体、可衡量、可达成、相关、有时间窗口。

### 4.1 产品目标

产品目标建议从“战略目标 -> 产品目标 -> 功能目标 -> 指标口径”逐层拆解。不要只写“提升效率”“增强体验”，而要明确谁的效率、哪个任务、提升多少、多久验证、由谁负责。

| 层级 | 说明 | 示例 |
|---|---|---|
| 战略目标 | 公司或部门层面的方向 | 提升区域经营管理效率，降低决策对人工报表的依赖 |
| 产品目标 | 本产品直接承担的结果 | 建立统一经营分析入口，支持异常识别、归因和行动闭环 |
| 功能目标 | 具体模块要达成的效果 | 首页 3 分钟内完成经营状态判断；异常页 10 分钟内定位主要影响维度 |
| 指标口径 | 如何量化和验收 | 分析任务完成率、异常闭环率、报表制作耗时、指标可信度 |

产品目标总表：

| 目标类型 | 目标描述 | 指标 | 基线 | 目标值 | 时间窗口 | Owner | 数据来源 |
|---|---|---|---:|---:|---|---|---|
| 业务目标 | 产品对业务结果的贡献，例如提升收入、降低成本、加快响应、减少风险 | 异常响应时长、行动闭环率、决策采纳率 |  |  |  | 业务 Owner | 业务系统 / 任务系统 / 会议记录 |
| 用户目标 | 用户是否愿意用、能否完成任务、是否信任结果 | WAU、核心任务完成率、用户满意度、留存率 |  |  |  | 产品 Owner | 埋点 / 调研 / 访谈 |
| 效率目标 | 是否减少重复劳动、缩短分析链路、提升自助能力 | 报表制作耗时、分析任务耗时、自助查询占比 |  |  |  | 产品 + 数据 Owner | 埋点 / 工单 / SQL 日志 |
| 质量目标 | 数据和分析结果是否准确、及时、可追溯 | 数据刷新成功率、空值率、口径一致率、AI 引用覆盖率 |  |  |  | 数据 Owner | 数据质量平台 / 日志 |
| 风险控制目标 | 是否降低越权、误导、合规、稳定性风险 | 越权事件数、高危漏洞数、审计覆盖率、P95 响应时间 |  |  |  | 技术 + 安全 Owner | 审计日志 / 监控 / 安全扫描 |

数据产品常用 KPI 设计维度：

| 维度 | 适用问题 | 指标示例 | 注意事项 |
|---|---|---|---|
| Adoption 使用采纳 | 用户是否开始使用产品 | 激活率、WAU、目标角色覆盖率、首次完成核心任务比例 | 不要只看登录，要看是否完成关键任务 |
| Engagement 使用深度 | 用户是否持续使用核心能力 | 人均核心操作次数、下钻率、收藏指标数、复用分析次数 | 避免把无意义点击当成价值 |
| Task Success 任务成功 | 用户是否完成目标任务 | 分析任务完成率、异常确认率、导出后行动创建率 | 需要定义“完成”的行为口径 |
| Efficiency 效率提升 | 是否节省时间和人力 | 报表制作耗时下降、手工取数次数下降、自助分析占比 | 需要上线前基线 |
| Trust 可信度 | 用户是否信任数据和结论 | 指标可信度评分、口径争议数、数据质量告警数 | 数据产品必须重点跟踪 |
| Business Impact 业务影响 | 是否带来业务结果变化 | 异常闭环率、决策采纳率、风险降低、收入/成本影响 | 需要避免把相关性误当因果 |
| Reliability 稳定性 | 产品是否可稳定运行 | API P95、可用性、任务成功率、刷新延迟 | 应作为护栏指标 |
| Compliance 合规与安全 | 是否满足权限和审计要求 | 越权访问数、敏感数据脱敏率、审计日志覆盖率 | 不达标时应阻断上线 |

KPI 字典模板：

| 字段 | 说明 | 示例 |
|---|---|---|
| 指标名称 | 简洁、业务可理解 | 异常闭环率 |
| 指标目的 | 为什么要跟踪这个指标 | 衡量平台是否把洞察转化为业务动作 |
| 计算公式 | 明确定义分子、分母、过滤条件 | 已关闭异常行动项数 / 已确认异常数 |
| 统计周期 | 日 / 周 / 月 / 季度 | 周 |
| 统计范围 | 哪些用户、组织、数据或场景纳入统计 | MVP 首批区域运营用户 |
| 基线来源 | 上线前如何获得当前值 | 历史台账、访谈估算、旧系统日志 |
| 目标值 | 到什么程度算成功 | MVP 上线 8 周后 >= 70% |
| 指标方向 | 越高越好 / 越低越好 / 区间最优 | 越高越好 |
| 数据来源 | 埋点、任务系统、业务库、监控、调研等 | action_items 表 + anomalies 表 |
| Owner | 谁负责解释和推动改善 | 业务运营 Owner |
| 复盘频率 | 多久看一次 | 每周产品运营例会 |
| 触发动作 | 指标异常后做什么 | 低于阈值时回访用户并检查任务流转阻塞 |

领先指标和滞后指标要同时设计：

| 类型 | 作用 | 数据产品示例 |
|---|---|---|
| 领先指标 | 提前判断产品是否在创造价值，可快速迭代 | 激活率、核心任务完成率、异常确认率、下钻率、自助查询占比 |
| 滞后指标 | 验证最终业务结果，通常需要更长周期 | 决策采纳率、异常闭环率、报表人力成本下降、指标争议减少、风险事件下降 |
| 护栏指标 | 防止为了追主要指标牺牲质量、安全或体验 | 数据准确率、P95 响应时间、越权事件数、用户投诉数、AI 无引用回答比例 |

目标设定示例：

| 目标类型 | 目标描述 | 指标 | 基线 | 目标值 | 时间窗口 | Owner | 数据来源 |
|---|---|---|---:|---:|---|---|---|
| 业务目标 | 缩短经营异常从发现到行动的周期 | 异常平均闭环时长 | 3 天 | 1 天以内 | MVP 上线后 8 周 | 区域运营 Owner | 异常中心 + 任务系统 |
| 用户目标 | 让首批管理层形成稳定使用习惯 | 周活跃目标用户覆盖率 | 0% | >= 70% | MVP 上线后 4 周 | 产品 Owner | 登录与页面埋点 |
| 效率目标 | 减少分析师重复制作经营报表时间 | 周报表制作耗时 | 20 小时 / 周 | <= 8 小时 / 周 | MVP 上线后 8 周 | BI Owner | 工时记录 + 访谈 |
| 质量目标 | 提升核心指标口径一致性和可信度 | 核心指标口径确认率 | 40% | >= 95% | 上线前 | 数据 Owner | 指标字典审批记录 |
| 风险控制目标 | 保证敏感数据访问合规可审计 | 关键操作审计覆盖率 | 0% | 100% | 上线前 | 技术 + 安全 Owner | 审计日志 |

目标评审时必须确认：

- 每个目标是否有业务 owner 和数据 owner。
- 是否有上线前基线；没有基线时，是否有基线采集计划。
- 指标是否能通过系统自动采集；如果依赖人工调研，频率和样本如何保证。
- 指标是否能指导行动；如果指标变差，团队知道要检查哪个流程或模块。
- 是否有护栏指标，防止为了追求使用量而牺牲数据质量、安全或用户信任。

### 4.2 北极星指标

北极星指标是产品最核心的价值度量，应该让团队知道“我们最希望用户反复获得的价值是什么”。它不一定等于公司 KPI，也不等于最容易采集的使用量指标。

一个好的北极星指标应满足：

| 判断标准 | 说明 |
|---|---|
| 代表核心用户价值 | 用户完成关键任务后，这个指标应该自然增长 |
| 能连接业务结果 | 指标提升应与效率、成本、风险、收入或决策质量相关 |
| 可被产品动作影响 | 团队能通过功能、体验、数据质量或运营动作改善它 |
| 可持续追踪 | 口径稳定，能按周 / 月复盘趋势 |
| 不容易被刷高 | 单纯增加点击、登录或无意义使用不能显著提升它 |
| 可拆解 | 能拆成用户、场景、功能、数据源等子指标定位问题 |

数据产品常见北极星指标候选：

| 候选指标 | 定义 | 适用产品 | 计算方式示例 | 配套护栏 |
|---|---|---|---|---|
| 决策采纳率 | 用户基于平台洞察形成决策或行动的比例 | 管理驾驶舱、经营分析平台 | 有行动记录或会议采纳标记的洞察数 / 被查看的关键洞察数 | 数据准确率、错误结论投诉数 |
| 分析任务完成率 | 用户能否在平台内完成目标分析任务 | 自助分析、指标平台、AI 问数 | 完成定义动作的任务数 / 发起任务数 | 任务耗时、用户满意度 |
| 指标可信度 | 核心指标口径一致、数据准确、可追溯 | 指标管理、数据治理、BI 平台 | 通过口径审批且质量达标的核心指标数 / 核心指标总数 | 刷新延迟、质量告警数 |
| 使用效率提升 | 完成同一分析任务的时间下降比例 | 报表自动化、运营分析工具 | (旧流程耗时 - 新流程耗时) / 旧流程耗时 | 结果准确率、人工返工率 |
| 异常闭环率 | 发现异常后完成归因、分派、处理和复盘的比例 | 异常中心、经营监控、风控运营 | 已关闭并复盘的异常数 / 已确认异常数 | 误报率、漏报率、闭环质量评分 |
| 自助分析占比 | 用户不依赖分析师即可完成查询和分析的比例 | 自助 BI、语义层、AI 数据助手 | 自助完成的分析请求数 / 总分析请求数 | 高风险查询拦截率、AI 引用覆盖率 |

北极星指标选择步骤：

1. 明确产品类型：经营决策、指标治理、自助分析、数据质量、AI 分析助手或行动闭环。
2. 列出用户核心任务：看数、查因、问数、定义指标、处理异常、生成行动、复盘。
3. 选择最能代表“用户获得价值”的行为结果，而不是中间点击。
4. 检查是否能被拆解成输入指标和护栏指标。
5. 和业务 owner 对齐目标值、统计周期和复盘机制。

北极星指标拆解模板：

| 北极星指标 | 一级拆解 | 二级拆解 | 可影响动作 |
|---|---|---|---|
| 异常闭环率 | 异常发现数 | 检测规则覆盖、刷新频率、数据质量 | 增加核心指标监控规则 |
| 异常闭环率 | 异常确认率 | 误报率、严重度、影响范围说明 | 优化告警阈值和解释文案 |
| 异常闭环率 | 行动创建率 | 负责人、截止时间、建议动作 | 在异常详情页强化创建行动入口 |
| 异常闭环率 | 按期关闭率 | 任务提醒、协作机制、复盘模板 | 增加提醒和升级机制 |
| 异常闭环率 | 复盘完成率 | 复盘字段、会议节奏、owner 责任 | 固化复盘流程和模板 |

不建议作为北极星指标的例子：

- 页面访问量：只能说明有人打开，不代表完成了业务任务。
- 登录用户数：容易被行政推动拉高，但不能证明产品价值。
- 报表数量：更多报表可能意味着信息架构混乱。
- AI 调用次数：调用多不代表答案可信或被采纳。
- 导出次数：导出多可能说明平台内分析能力不足。

北极星指标最终写法：

```text
本产品的北极星指标是：[指标名称]。
选择原因：[它如何代表核心用户价值]，并且与 [业务目标] 直接相关。
计算口径：[分子 / 分母 / 统计范围 / 排除条件]。
目标值：[基线] -> [目标值]，在 [时间窗口] 内达成。
配套护栏指标：[数据质量]、[性能]、[权限/合规]、[用户满意度]。
复盘机制：[每周 / 每月] 由 [owner] 牵头复盘，并输出改进行动。
```

成功标准分层：

| 阶段 | 成功标准 | 示例 |
|---|---|---|
| Alpha / 原型 | 验证问题和方案方向是否成立 | 5-8 位核心用户试用后，80% 认可核心流程有价值 |
| Beta / MVP | 验证核心任务能否闭环 | 首批用户核心任务完成率 >= 70%，关键数据准确率 >= 95% |
| GA / 上线 | 验证稳定性、权限、安全和业务可用性 | P95 < 2.5s，审计覆盖 100%，无高危漏洞 |
| 上线后 4-8 周 | 验证持续使用和业务结果 | WAU 覆盖率 >= 70%，异常闭环率 >= 60%，报表耗时下降 40% |
| 上线后季度复盘 | 验证是否值得继续投入 V1 / V2 | 北极星指标持续改善，业务 owner 确认扩大范围 |

目标与成功标准评审 Checklist：

- 是否有一个清晰的北极星指标，而不是一堆同等重要的数字？
- 是否同时覆盖业务结果、用户行为、效率、质量、风险和技术稳定性？
- 是否为每个 KPI 定义了公式、基线、目标值、时间窗口、owner 和数据来源？
- 是否区分领先指标、滞后指标和护栏指标？
- 是否避免使用无法指导行动的虚荣指标？
- 如果 KPI 未达标，是否知道应该检查哪个用户旅程、功能模块或数据链路？
- 是否有固定复盘节奏，把 KPI 变成产品迭代输入？

1. 用户登录并进入首页总览。

5. 用户查看贡献度、趋势、对比和可能原因。
6. 用户生成行动项，分派给负责人。
7. 负责人反馈处理进展。
8. 系统追踪指标恢复情况并生成复盘记录。

## 9. 数据需求

数据需求是数据产品 PRD 的核心章节，用于说明产品依赖哪些数据、这些数据是否可信、如何定义指标、如何控制质量、如何授权访问，以及上线前必须满足哪些数据验收条件。

这一章建议至少覆盖三类内容：

- 数据源清单：回答数据从哪里来、谁负责、多久刷新、怎么接入、能否稳定使用。
- 指标字典模板：回答指标叫什么、怎么算、谁认可、在哪些维度和场景下使用。
- 数据质量规则：回答什么叫数据可用、质量失败如何处理、是否影响上线或展示。

数据需求不是后端或数据工程团队的内部文档，它应该被业务 owner、产品、数据、研发、测试、安全共同评审。尤其是用于 AI 问数、经营决策、自动告警或行动闭环的产品，必须在 PRD 阶段就明确数据来源、口径、血缘、权限和质量门禁。

| 分类 | 说明 | 示例 |

| 过程指标 | 反映业务链路中间过程 | 线索数、转化率、响应时长 |
| 质量指标 | 反映数据或业务质量 | 数据准确率、口径一致率、退单率 |
| 风险指标 | 反映异常、合规或损失风险 | 越权访问数、逾期任务数、异常未处理数 |
| 产品使用指标 | 反映产品自身使用情况 | WAU、任务完成率、下钻率 |

指标口径变更管理：

| 变更项 | 要求 |
|---|---|
| 变更原因 | 必须说明业务原因、影响范围和是否追溯历史 |
| 影响评估 | 标注影响哪些页面、报表、API、AI 问数、告警规则和下游系统 |
| 审批流程 | 业务 owner、数据 owner、产品 owner 必须确认 |
| 版本管理 | 保留旧版本口径和生效日期，支持历史查询解释 |

指标验收 Checklist：

- 指标是否有唯一编码和明确业务名称。
- 指标定义是否被业务 owner 确认。
- 计算公式是否明确分子、分母、过滤条件、粒度和维度。
- 是否指定 source of truth，避免多个来源相互冲突。
- 是否有刷新频率、延迟要求和质量规则。
- 是否有版本、生效日期和废弃策略。
- 是否能被测试用例验证，而不是只存在文档中。

### 9.3 数据质量规则

数据质量规则用于定义“什么样的数据可以被展示、分析、告警、用于 AI 回答或用于业务决策”。数据质量应覆盖完整性、唯一性、有效性、及时性、准确性、一致性和适用性。

数据质量维度：

| 维度 | 定义 | 示例规则 | 失败影响 |
|---|---|---|---|
| Completeness 完整性 | 必要字段是否存在且不为空 | order_id、amount、dt 不可为空 | 指标缺失、统计偏差 |
| Uniqueness 唯一性 | 主键或业务键是否重复 | order_id 重复率 = 0 | 重复计数、收入虚高 |
| Validity 有效性 | 字段是否符合类型、格式、枚举或范围 | conversion_rate 在 0-100% | 展示异常、计算错误 |
| Timeliness 及时性 | 数据是否在 SLA 时间内到达 | T+1 09:00 前完成刷新 | 决策延迟、页面过期 |
| Accuracy 准确性 | 数据是否与权威来源或对账结果一致 | ERP 收入与财务总账差异 < 0.5% | 决策错误、信任下降 |
| Consistency 一致性 | 跨表、跨系统、跨层级是否逻辑一致 | 区域收入汇总 = 全国收入 | 口径争议、汇总错误 |
| Fitness for Purpose 适用性 | 数据是否满足当前业务场景 | 行动闭环分析必须有 owner 和状态字段 | 数据存在但无法支持决策 |

数据质量规则总表：

| 规则ID | 对象 | 维度 | 规则 | 阈值 | 检查频率 | 严重级别 | 失败处理 | Owner |
|---|---|---|---|---|---|---|---|---|
| DQ-001 | 核心指标 | 完整性 | 核心指标值不可为空 | 空值率 < 1% | 每次刷新 | P0 | 阻断发布 / 告警 | Data Owner |
| DQ-002 | 订单数据 | 唯一性 | order_id 不可重复 | 重复率 = 0 | 每次刷新 | P0 | 阻断收入指标计算 | Data Engineer |
| DQ-003 | 数据刷新 | 及时性 | 核心链路必须按 SLA 完成 | T+1 09:00 前 | 每日 | P0 | 页面标记延迟，通知 owner | Platform |
| DQ-004 | 维表映射 | 完整性 | 组织、区域、产品映射不可缺失 | 未映射比例 < 0.5% | 每日 | P1 | 进入待修复队列 | Data Steward |
| DQ-005 | 收入指标 | 准确性 | 与财务对账差异可控 | 差异 < 0.5% | 每日 / 月结 | P0 | 阻断管理层口径展示 | Finance Owner |
| DQ-006 | 行为事件 | 有效性 | event_name 必须在埋点字典内 | 非法枚举 = 0 | 小时级 | P1 | 丢弃非法事件并告警 | Product Analyst |
| DQ-007 | AI 问数 | 适用性 | AI 回答必须引用已批准指标 | 引用覆盖率 100% | 每次回答 | P0 | 阻断回答或降级为无法回答 | AI Owner |

质量失败分级：

| 级别 | 定义 | 处理要求 | 示例 |
|---|---|---|---|
| P0 阻断 | 会影响管理决策、财务口径、权限合规或 AI 输出可信度 | 阻断发布或隐藏相关模块，立即通知 owner | 收入对账失败、核心指标为空、越权数据出现 |
| P1 严重 | 影响部分页面、维度或分析结果，但可降级展示 | 页面提示、告警、进入修复队列 | 部分区域映射缺失、埋点延迟 |
| P2 一般 | 影响较小或可在后台修复 | 记录日志，定期修复 | 非核心字段格式异常 |
| P3 提示 | 不影响业务结果，但影响治理完善度 | 纳入治理 backlog | 描述缺失、owner 未补充 |

质量失败处理流程：

1. 检测：调度任务、数据质量平台或实时校验发现异常。
2. 定级：根据影响范围、指标重要性、用户场景和合规风险确定 P0-P3。
3. 降级：对受影响页面、指标、API、AI 回答或告警规则执行隐藏、提示、缓存或阻断。
4. 通知：自动通知 data owner、system owner、product owner 和相关业务方。
5. 修复：定位源头、修复数据、回补历史、重新计算指标。
6. 复盘：记录原因、影响、恢复时间和预防措施。

质量监控与告警模板：

| 监控项 | 指标 | 阈值 | 告警对象 | 展示策略 | 恢复条件 |
|---|---|---|---|---|---|
| 数据刷新 | refresh_delay_minutes | > 60 min | Platform + Data Owner | 页面显示“数据延迟” | 最新分区刷新成功 |
| 空值率 | null_rate | > 1% | Data Steward | 隐藏受影响维度 | 空值率恢复阈值内 |
| 重复率 | duplicate_rate | > 0 | Data Engineer | 阻断指标计算 | 去重后重新跑数 |
| 对账差异 | reconciliation_diff | > 0.5% | Finance + Data Owner | 阻断财务指标展示 | 对账通过 |
| AI 引用 | citation_coverage | < 100% | AI Owner | 阻断回答 | 回答包含有效引用 |

### 9.4 数据权限、隐私与合规

数据需求必须明确数据可见范围，尤其是客户信息、个人信息、财务数据、合同数据、组织绩效和 AI 输入输出。

| 数据类别 | 示例字段 | 敏感等级 | 访问原则 | 处理要求 |
|---|---|---|---|---|
| 个人信息 | name, phone, email, user_id | Restricted | 最小必要、按角色授权 | 脱敏、加密、审计 |
| 财务数据 | revenue, margin, refund_amount | Confidential | 按组织和职责授权 | 行级权限、导出审批 |
| 客户数据 | customer_name, contract_id | Confidential | 业务相关人员可见 | 脱敏、访问日志 |
| 运营指标 | conversion_rate, order_count | Internal | 授权角色可见 | 口径说明、下载控制 |
| AI 上下文 | prompt, retrieved_context, answer | Confidential | 按原始数据权限继承 | 禁止越权召回、保留审计 |

权限与审计要求：

- 所有数据查询必须经过用户身份、角色和数据范围校验。
- 敏感字段默认脱敏，只有审批角色可查看明文。
- 导出、批量下载、AI 问数、指标口径变更、权限变更必须写审计日志。
- AI/RAG 场景必须继承原始数据权限，不允许因为向量化或缓存绕过权限。
- 数据保留周期、删除策略和跨境传输要求必须在上线前确认。

## 10. 技术实现框架

### 10.1 推荐架构原则

- 优先采用模块化单体或清晰分层架构，避免早期过度微服务化。
- 数据产品必须把指标口径、权限、审计、数据质量作为一等能力。
- 面向分析场景，读多写少，优先设计聚合、缓存和异步任务。
- 对 AI 能力保持可替换：模型调用、提示词、结果版本、人工确认都要留痕。

### 10.2 方案 A：保守快上线

| 维度 | 建议 |
|---|---|
| 前端 | React / Vue + TypeScript |
| 后端 | Python FastAPI / Java Spring Boot |
| 数据库 | PostgreSQL / MySQL |
| 分析查询 | PostgreSQL 聚合表 / ClickHouse 视数据量决定 |
| 任务调度 | Cron / Airflow / Prefect |
| 缓存 | Redis |
| BI 图表 | ECharts / AntV / Recharts |
| 部署 | Docker + 单体服务 + 定时任务 |
| 适用场景 | 团队小、3 个月内上线、数据规模中等、需求仍在探索 |

### 10.3 方案 B：可演进数据平台

| 维度 | 建议 |
|---|---|
| 前端 | Next.js / React + TypeScript |
| 后端 | FastAPI / Spring Boot + 模块化领域边界 |
| 数据库 | PostgreSQL + ClickHouse / DuckDB / BigQuery / Snowflake |
| 数据建模 | dbt / SQL model layer |
| 调度编排 | Airflow / Dagster / Prefect |
| 消息 / 事件 | Kafka / Pulsar / Cloud PubSub |
| 语义检索 | pgvector / Milvus / Elasticsearch |
| 可观测性 | OpenTelemetry + 日志 + 指标 + 链路追踪 |
| 部署 | Kubernetes / Cloud Run / Container Apps |
| 适用场景 | 多团队协作、数据量增长快、要做指标治理和 AI 分析 |

### 10.4 技术方案评分

| 维度 | 权重 | 方案 A | 方案 B | 说明 |
|---|---:|---:|---:|---|
| 交付速度 | 20 | 5 | 3 | A 更快上线 |
| 可维护性 | 20 | 3 | 4 | B 边界更清晰 |
| 可扩展性 | 20 | 3 | 5 | B 更适合长期扩展 |
| 成本 | 15 | 4 | 3 | A 初期成本低 |
| 运维风险 | 15 | 4 | 3 | A 组件少，B 治理要求高 |
| 安全合规 | 10 | 3 | 4 | B 更易内建审计和治理 |
| 加权总分 | 100 | 375 | 375 | 若 3 个月内上线选 A；若确定长期平台化选 B |

推荐：MVP 按方案 A 落地，但模块边界、数据模型和权限审计按方案 B 预留。这样既不拖慢首发，也不会把未来的平台化能力堵死。

## 14. 指标体系与埋点需求

### 14.1 产品使用指标

| 指标 | 定义 | 用途 |
|---|---|---|
| DAU / WAU | 日 / 周活跃用户数 | 判断使用规模 |
| 核心任务完成率 | 完成核心分析或闭环任务的比例 | 判断是否真正有用 |
| 下钻率 | 从总览进入详情或归因的比例 | 判断洞察吸引力 |

## 16. 安全、权限与合规

- 权限模型：RBAC + 数据范围控制，必要时增加行级 / 列级权限。
- 敏感数据：个人信息、财务数据、客户数据需要脱敏、加密或最小化展示。
- 审计要求：登录、导出、权限变更、指标变更、AI 问数、审批动作必须留痕。
- 数据保留：定义明细数据、聚合数据、日志、模型输出的保留周期。
- AI 合规：AI 输出必须可追溯到数据来源，不允许直接生成不可验证结论。
- 跨境与外部传输：如涉及跨境、第三方 API、云服务，需要法务和安全评审。

## 17. 风险与约束

| 风险 | 影响 | 触发条件 | 缓解方案 | owner |
|---|---|---|---|---|
| 指标口径不一致 | 用户不信任平台 | 同一指标多个版本 | 建立指标 owner、版本和审批 |
| 数据质量不足 | 分析结论错误 | 缺失、重复、延迟 | 数据质量门禁和告警 |
| 权限设计不足 | 合规风险 | 用户看到越权数据 | 权限测试和审计 |
| 需求范围膨胀 | 延期 | MVP 加入过多功能 | P0/P1/P2 严格分层 |
| 性能不足 | 用户流失 | 查询慢、图表卡顿 | 聚合表、缓存、异步任务 |
| AI 输出不可靠 | 错误决策 | 无引用、幻觉 | 强制引用、人工确认、灰度上线 |

## 21. PRD 评审 Checklist

- 背景是否说明清楚为什么现在必须做。
- 目标是否有可量化指标和基线。
- 首批用户和核心场景是否明确。
- MVP / V1 / V2 是否分层清楚。
- Out of Scope 是否足够明确。
- 数据源、指标口径、刷新频率是否明确。
- 权限、审计、敏感数据是否覆盖。
- 技术方案是否与团队能力和上线时间匹配。
- 验收标准是否可测试、可演示、可度量。
- Open Questions 是否有 owner 和截止时间。
